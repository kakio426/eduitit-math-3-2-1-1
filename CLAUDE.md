## 기본 원칙

- 차시별 게임은 독립 실행 가능한 단일 HTML 패키지를 우선합니다.
- 학생 대상은 태블릿 가로와 컴퓨터 화면입니다.
- 스마트폰 세로 화면은 기본 지원 범위가 아닙니다.
- 첫 화면은 제목, 한 줄 목표, 시작 버튼이 가장 먼저 보여야 합니다.
- 첫 화면과 마지막 결과 화면은 이미지 생성 기반 RasterStage로 제작하고, 매 판 달라지는 텍스트·점수·버튼은 HTML 오버레이로 얹습니다.
- 모든 차시는 `16:10` Stage, 기준 제작 크기 `1280×800`을 지킵니다. `index.html`의 `<main class="game">`에는 `data-stage-ratio="16:10"`과 `data-stage-size="1280x800"`을 두고, `.stage-shell`이 contain 폭과 비율을 담당하게 합니다.
- 첫 화면 커버 표준은 `data-cover-standard="generated-title-overlay"`입니다. `cover-generated.webp`는 글자 없는 대표 장면 배경으로 `.raster-bg`에 `object-fit: cover`로 깔고, 게임명은 생성형 이미지 제목 자산(`title-*-generated.webp`)을 `.hero-title-art`로 얹으며, 한 줄 목표와 `시작` 버튼은 보이는 HTML 오버레이로 둡니다.
- `cover-generated.webp` 한 장 안에 제목·목표·시작 버튼을 baked-in 하거나, 투명 `cover-start-hitbox`로 시작시키는 방식은 새 차시에 쓰지 않습니다. 기존 1차시처럼 아직 이전 방식인 화면만 `data-cover-standard="legacy-raster-poster"`로 명시하고, 복제 기준으로 삼지 않습니다.
- `소리` 같은 전역 조작 버튼은 Stage 밖 viewport에 fixed로 띄우지 않고, `.stage-shell` 안의 상단 오른쪽 보조 슬롯에 작게 둡니다. `.stage-shell`의 `--sound-button-size`, `--sound-gap`, `--sound-reserve`로 위치와 예비 공간을 고정하고, `.sound-toggle`에는 화면별 `transform`/active-screen별 위치 보정을 쓰지 않습니다. `top-row`/`hud`는 같은 `--sound-reserve`만큼 비워 버튼이 배지·문제·선택지를 가리지 않게 합니다.
- 소리 버튼은 텍스트 pill이 아니라 `width/height: var(--sound-button-size)`인 원형 SVG 아이콘 버튼으로 만듭니다. 화면에 `소리` 글자를 직접 노출하지 말고, 켜짐/꺼짐은 SVG 파형과 `aria-label`로만 표현합니다.
- 상단 배지·단원 pill·소리 버튼은 같은 기준선에 놓습니다. `.top-row`는 `top/left/right`와 `height: var(--sound-button-size)`, `gap: var(--sound-gap)`을 명시하고 `inset` 축약을 쓰지 않습니다. 문제 화면 `.hud`는 `align-items: start`와 `min-height: var(--sound-button-size)`를 둡니다.
- 브랜드/단원/상태 배지는 내용보다 오른쪽으로 늘어나면 실패입니다. `flex: 0 0 auto`, `width: fit-content`, `max-width: max-content`, 작은 좌우 패딩 변수(`--top-control-pad-x`)와 gap 변수(`--top-control-icon-gap`)를 쓰고, 안정 슬롯이 꼭 필요하지 않으면 `min-width`로 빈 공간을 만들지 않습니다.
- 차시를 만들거나 화면을 크게 바꾼 뒤에는 루트에서 `node scripts/check-stage-ratio.mjs`를 실행해 Stage 비율 계약을 통과해야 합니다.
- 문제와 선택지는 장식보다 항상 우선합니다.
- 보상은 하나의 중심 구조로 유지합니다. 여러 보상 체계를 동시에 전면 배치하지 않습니다.
- 문구는 초등학생이 소리 내어 읽어도 자연스럽게 씁니다.
- 첫 화면 제목은 단순 큰 글자가 아니라 **타이포그래픽 포스터/타이틀 아트**로 다룹니다. 사용자가 `그림으로`, `GPT Image`, `제목 이미지`를 요구하면 기존 첫 화면 배경은 유지하고, 제목 부분만 독립 래스터 오버레이(`title-logo-generated.webp` 또는 `title-poster-generated.webp` 같은 배포 자산)로 생성해 얹습니다. 전체 커버 이미지를 갈아엎거나 HTML/CSS/SVG로 어설프게 흉내 내지 않습니다. 접근성용 실제 제목은 `visually-hidden`으로 남깁니다.
- 첫 화면 제목 자산은 반드시 GPT Image/imagegen 같은 생성형 이미지 산출물이어야 합니다. 로컬 폰트, Pillow, canvas, SVG, CSS text-shadow로 만든 "그림처럼 보이는 텍스트"는 실패입니다. 생성 원본(`title-*-source.png` 또는 `title-*-chromakey.png`)과 배경 제거 PNG/WebP를 함께 보관합니다.

## 초3 문제 화면 과밀 금지 (강제)

앞으로 모든 차시의 문제 화면은 "초등학교 3학년 학생이 3초 안에 지금 무엇을 해야 하는지 알 수 있는가?"를 통과해야 합니다. 통과하지 못하면 완성 화면이 아니라 실패 화면으로 보고 다시 줄입니다.

- 문제 화면의 목적은 **현재 한 단계의 수학 행동**을 하게 하는 것입니다. 개념 설명판, 풀이 해설판, 힌트판, 계산 미리보기, 보상 상태판을 한 화면에 동시에 펼치지 않습니다.
- 한 문제 화면에 기본으로 보여도 되는 학습 요소는 4개뿐입니다: **큰 문제**, **현재 단계 조작판**, **한 줄 지시문**, **선택지**. 나머지는 접거나, 완료 뒤에 작은 체크칩으로 줄이거나, 힌트 버튼 뒤로 숨깁니다.
- 설명 문구는 기본 **한 줄**입니다. 문제 화면에서 문장형 해설을 여러 줄 쓰지 않습니다. 예: `먼저 63×3을 골라요`, `7과 곱한 값을 골라요`. 교사용 설명처럼 긴 문장은 금지합니다.
- 원리는 글로 설명하지 말고 **보이는 조작**으로 드러냅니다. 자리값, 0 붙이기, 내림, 나머지, 검산은 블록·칸·스티커·화살표처럼 학생이 바로 볼 수 있는 구조로 보여 줍니다.
- 단계형 문제는 **현재 단계만 크게** 보여 줍니다. 이전 단계는 작은 완료 칩으로 접고, 다음 단계는 `?` 또는 잠금 상태로 둡니다. 여러 단계의 계산판을 옆에 동시에 펼치지 않습니다.
- 선택지는 항상 가장 눈에 잘 띄는 한 덩어리로 둡니다. 선택지와 큰 문제 사이에 장식, 보상판, 긴 설명 패널을 끼워 넣지 않습니다.
- 문제·현재 단계·선택지가 Stage의 대부분을 차지해야 합니다. 보조 정보(진행도, 콤보, 보상 상태, 소리 버튼)는 작게 두고, 풀이 판단을 방해하면 숨깁니다.
- 힌트는 기본으로 열어 두지 않습니다. 학생이 눌렀을 때도 **지금 단계 힌트 1개**만 보여 주고, 전체 풀이 과정을 한 번에 펼치지 않습니다.
- 학생 화면에 `핵심 메커니즘`, `메타 필드`, `부분곱`, `정답 수 게이트` 같은 제작자 용어를 쓰지 않습니다. 수학 용어가 필요하면 교과서식 짧은 말로 바꿉니다.
- 스크린샷을 보고 "문제보다 패널이 많다", "읽어야 할 문장이 많다", "지금 눌러야 할 것이 바로 안 보인다"는 느낌이 들면 무조건 덜어냅니다.

## 핵심 설계 철학 (모든 차시 공통)

이 작업실이 만드는 모든 게임은 아래 철학을 따릅니다. 이것은 차시별 소재가 바뀌어도 변하지 않는, 이 자료의 정체성입니다. 새 차시를 만들 때마다 문제 생성·보상 설계·결과 구조가 이 철학에 맞는지 먼저 확인합니다.

### 1. 계산은 "행동의 도구"다

문제는 그 자체로 끝나는 반복 과제가 아니라, 점수·등급·결과 보상으로 이어지는 행동의 도구입니다. 학생은 문제를 맞혀야 다음 보상(상자·연료·부품·바구니 등 차시 소재)을 열 수 있고, 그 결과에 따라 보상이 크게 오르내립니다. 계산은 예측 가능한 영역이지만 보상은 예측 불가능하게 설계해, 학생은 같은 유형의 문제를 풀면서도 매번 다른 결과를 경험합니다.

교육적으로는 반복 연습에 게임의 간헐적 보상을 결합한 구조입니다. 학생은 정답을 맞히는 데서 멈추지 않고, 다음 보상과 최종 결과를 기대하며 더 많은 문제에 자발적으로 접근합니다. 이때 무작위성은 학습을 흐리는 장치가 아니라, 이미 배운 계산 절차를 더 자주 사용하게 만드는 동기 장치로 작동합니다.

### 2. 무작위성은 반드시 유지한다

랜덤 보상은 빼지 않습니다. 아이들이 흥미를 가지고 다시 도전하게 만드는 핵심 동력입니다. 단, 무작위성은 "계산을 안 해도 되는 핑계"가 아니라 "이미 한 계산의 결과를 매번 새롭게 만드는 장치"여야 합니다. 보상에는 기대감과 긴장감이 함께 있어야 합니다(잘 풀어도 가끔 손해, 운이 따르면 한 번에 도약).

### 3. 정답이 전부를 보장하지 않는다

- 좋은 것은 얻기 어렵게 만듭니다. 한 판으로 최고 등급에 쉽게 닿으면 반복할 이유가 사라집니다.
- 정답을 많이 맞히는 것은 분명히 유리해야 합니다. 하지만 그것만이 모든 걸 쉽게 얻는 유일한 통로가 되어서는 안 됩니다.
- 못하는 아이도 무언가는 반드시 얻습니다(빈손 금지). 동시에 운이 따르면 평소보다 더 좋은 것을 얻을 수도 있습니다.
- 그래서 잘하는 아이는 실력으로, 아직 서툰 아이는 실력+운으로 위를 노립니다. 누구도 완전히 막히지 않고, 누구도 한 번에 다 갖지 않습니다.

### 4. "다음엔 더 멀리"를 만든다

결과는 차시마다 자체 완결형 등급으로 보여 줍니다. 학생이 "이번엔 여기까지 왔지만 다음엔 더 멀리 갈 수 있다"고 느끼게 만드는 것이 이 자료의 핵심입니다. 한 개의 콘텐츠를 여러 번 반복하게 만드는 재도전 동기가 모든 차시 설계의 최종 점검 기준입니다.

### 차시 설계 점검 질문

새 차시를 만들 때 아래에 모두 "그렇다"가 나와야 합니다.

- 문제를 푸는 것이 보상으로 이어지는 "행동의 도구"로 설계되었는가?
- 랜덤 보상이 있고, 기대감과 긴장감이 함께 있는가?
- 최고 보상은 충분히 얻기 어려운가? (한 판으로 쉽게 닿지 않는가)
- 정답이 유리하되, 정답만이 모든 걸 쉽게 주는 유일한 통로는 아닌가?
- 못해도 빈손은 아니고, 운으로 한 단계 더 갈 여지가 있는가?
- 학생이 "다음엔 더 멀리"를 떠올리며 다시 하고 싶어지는가?

### RasterStage 화면 규칙

- 모든 새 차시의 첫 화면과 마지막 결과 화면은 이미지 생성으로 만든 대표 장면을 배경으로 사용합니다.
- 모든 RasterStage와 주요 게임 화면은 `1280×800` 기준 `16:10` 안에서 설계합니다. 생성 이미지가 다른 비율이어도 Stage 안에서는 `object-fit: cover/contain`과 HTML 오버레이로 맞추고, Stage 자체 비율을 바꾸지 않습니다.
- 첫 화면 배경 이미지는 글자 없는 대표 장면이어야 합니다. 게임명·한 줄 목표·시작 버튼을 한 이미지에 구워 넣지 않고, `cover-generated.webp` 배경 + `.hero-title-art` 제목 이미지 + HTML 목표/버튼 오버레이로 나눕니다.
- 시작 버튼은 학생에게 보이는 실제 HTML 버튼입니다. 커버 전체에 투명 hitbox를 얹는 방식은 legacy 화면에만 남기고 새 차시에서는 금지합니다.
- 첫 화면 제목은 배경 이미지 위에 별도의 타이틀 아트로 얹을 수 있습니다. 이때 **생성 대상은 전체 커버가 아니라 제목 로고만**입니다. 기존 `cover-generated.webp` 같은 대표 장면을 유지하고, GPT Image 등으로 만든 제목 로고를 초록 배경 제거 또는 투명 PNG/WebP로 분리해 `.hero-title-art`로 올립니다.
- 학생에게 보이는 제목은 하나의 그림처럼 보여야 합니다. HTML/CSS로 급조한 밋밋한 큰 글자, 어설픈 SVG 로고, 전체 커버를 새로 합성해 장면 구도를 망가뜨리는 방식, AI가 철자를 틀린 한글 이미지는 모두 실패로 봅니다.
- 제목 로고를 만드는 순간에는 `image_gen`/GPT Image 등 생성형 이미지 도구를 실제로 사용합니다. 로컬 폰트로 직접 래스터화하거나 CSS 효과를 캡처해 만든 파일은 `.hero-title-art`에 연결하지 않습니다.
- GPT Image 등 생성형 이미지로 제목을 만들 때는 `standalone title/logo asset`, `exact Korean text`, `no full scene`, `flat chroma-key background` 조건으로 만들고, 한글 철자를 화면 캡처로 검수합니다. 철자가 틀리거나 품질이 맞지 않으면 같은 제목 오버레이 슬롯에서 재생성합니다.
- 결과 화면 생성 이미지는 차시의 중심 보상 장면과 매스몬 동행 캐릭터가 자연스럽게 들어간 장면으로 만들고, 최종 점수·정답 수·칭찬 문구·다시하기 버튼처럼 매 판 달라지는 요소는 HTML로 얹습니다.
- 생성 원본 PNG는 작업실에 보관하고, 학생에게 배포하는 `index.html`에서는 WebP 같은 경량 포맷을 우선 사용합니다.
- 문제 화면과 선택지 화면은 생성 이미지보다 읽기 쉬운 HTML/CSS UI를 우선합니다. 장면 이미지는 문제와 선택지를 가리지 않는 배경 또는 보상 연출에만 씁니다.
- `README.md`와 `REPORT.md`에는 사용한 생성 이미지 파일명, 배치 의도, 동적 HTML 오버레이 범위를 기록합니다.

| 1 | 들이 비교와 L, mL | 매스몬 물통 채우기 시합 | 눈금을 읽고 알맞은 들이 고르기 | 물통 등급 |

### 6단원 자료의 정리(그림그래프) — 영역: 자료와 가능성 / 성취기준 `[4수04-XX] (자료 정리·그래프)`

자료를 표와 그림그래프로 정리·표현하고, 그래프를 읽어 해석한다.

| 차시 | 배움주제 | 게임명(가안) | 핵심 메커니즘 | 중심 보상 |

| 1 | 자료를 표로 정리하기 | 매스몬 인구조사 | 자료를 세어 표 칸 채우기 | 완성 표 등급 |
| 2 | 그림그래프 읽기 | 매스몬 그림 단위 해독 | 그림 1개의 단위로 값 읽어 답하기 | 해독 등급 |
| 3 | 그림그래프로 나타내기 | 매스몬 그림 도장 찍기 | 큰 그림·작은 그림 수를 맞춰 찍기 | 그래프 등급 |

## 배포 (GitHub Pages)

학생용 정적 HTML 게임은 이 레포지토리 하나의 GitHub Pages로 공유합니다(별도 공개 레포 금지). 필요한 게임 폴더만 `actions/upload-pages-artifact`와 `actions/deploy-pages`로 배포하고, 공유 전 `curl -I -L <url>`로 실제 URL이 `HTTP 200`인지 확인합니다. Pages 실패 시 `gh run view <run-id> --log-failed`로 원인을 먼저 확인합니다. checkout 단계에서 submodule 오류가 나면 `.gitmodules`와 `git ls-files -s`로 깨진 `160000` gitlink를 확인하고, 필요 없는 임시 gitlink는 `git rm <path>`로 제거합니다.
