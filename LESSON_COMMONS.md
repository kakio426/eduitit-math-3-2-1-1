# 매스몬 차시 공통 빌더 체크리스트

이 문서는 1-2-3차시에서 굳어진 공통 제작 습관을 새 차시에 바로 적용하기 위한 실무 체크리스트입니다. 큰 원칙과 로드맵은 `AGENTS.md`/`CLAUDE.md`, 화면 계약은 `SERIES_CONTRACT.md`, 제작 절차는 `.claude/skills/eduitit-mathmon-lesson/SKILL.md`를 기준으로 둡니다.

## 1. Stage와 첫 화면

- 모든 차시는 `16:10` Stage와 기준 크기 `1280x800`을 지킵니다.
- `index.html`의 `<main class="game">`에는 `data-stage-ratio="16:10"`과 `data-stage-size="1280x800"`을 둡니다.
- `.stage-shell`이 contain 폭과 `aspect-ratio: 16 / 10`을 담당하고, 각 `.screen`은 Stage 안을 채웁니다.
- 화면을 크게 바꾼 뒤에는 루트에서 `node scripts/check-stage-ratio.mjs`를 실행합니다.
- 첫 화면은 `generated-title-overlay` 표준을 씁니다. 글자 없는 `cover-generated.webp`를 배경으로 깔고, 생성형 이미지 제목 자산을 `.hero-title-art`로 얹습니다.
- 제목 자산은 GPT Image/imagegen 같은 생성형 이미지 산출물이어야 합니다. HTML/CSS/SVG/로컬 폰트로 만든 글자 이미지는 제목 아트로 쓰지 않습니다.
- 한 줄 목표는 실제 HTML 오버레이로 둡니다. 새 차시와 이관 완료 차시는 `data-cover-start-standard="generated-button-art"`를 붙이고, `start-button-generated.webp` 같은 생성형 버튼 아트를 실제 HTML 버튼 안에 넣습니다.
- 시작 버튼은 1차시 포스터형 버튼처럼 플레이 아이콘이 들어간 두툼한 노란 래스터 버튼을 기준으로 삼고, 1280×800 Stage 기준 `400-460px × 140-170px` 정도로 목표 바로 아래에 둡니다. 커버 전체 투명 hitbox는 새 차시에 쓰지 않습니다.
- 기존 `generated-title-overlay` 차시의 `.primary-button` 시작은 개별 이관 전까지 `compatibility-primary-button`으로 보존할 수 있습니다. 이 호환 상태는 새 차시 기준이 아니며, 이관할 때는 생성형 버튼 자산과 실제 `.cover-start-button` 구조로 바꿉니다.

## 2. 상단 슬롯과 소리 버튼

- `소리 버튼`은 `.stage-shell` 안의 상단 오른쪽 보조 슬롯에 둡니다.
- `.stage-shell`에는 `--sound-button-size`, `--sound-gap`, `--sound-reserve`를 두고, `.top-row`와 `.hud`가 같은 예비 공간을 비웁니다.
- `.sound-toggle`은 원형 SVG 아이콘 버튼입니다. 화면에 `소리` 글자를 직접 보이지 않게 하고, 켜짐/꺼짐은 SVG와 `aria-label`로 표현합니다.
- `.top-row`는 `top`, `left`, `right`, `height`, `gap`을 명시합니다. 화면별 `transform`, active-screen별 위치 보정, viewport fixed 배치는 쓰지 않습니다.
- 배지는 `flex: 0 0 auto`, `width: fit-content`, `max-width: max-content`를 기본으로 하며, 불필요한 `min-width`로 빈 칸을 만들지 않습니다.

## 3. 문제 화면 3초 규칙

- 문제 화면은 초3 학생이 `3초` 안에 지금 할 일을 알 수 있어야 합니다.
- 기본으로 펼쳐 둘 학습 요소는 큰 문제, 현재 단계 조작판, 한 줄 지시문, 선택지뿐입니다.
- 이전 단계는 작은 완료 칩으로 접고, 다음 단계는 `?`나 잠금 상태로 둡니다.
- 정답을 고른 뒤에는 선택지 색만 바꾸고 바로 넘기지 않습니다. 계산판·칸·블록이 정답값으로 바뀌고, 짧은 확인 문구가 잠깐 보인 뒤 다음 단계로 갑니다.
- 마지막 단계 뒤에는 보상 모달이 뜨기 전에 완성식이나 완성값을 먼저 보여 줍니다. 자동으로 모달을 덮기보다 학생이 `바람 보기`, `상자 열기`처럼 차시 소재에 맞는 버튼을 눌러 보상으로 넘어가는 흐름을 우선합니다. 모달이 정답 확인을 가리면 실패입니다.
- 힌트는 기본으로 닫아 두고, 열어도 지금 단계 힌트 1개만 보입니다.
- 원리는 긴 설명보다 블록, 칸, 스티커, 화살표처럼 바로 보이는 조작으로 보여 줍니다.
- 선택지와 큰 문제 사이에 장식, 보상판, 긴 설명을 끼우지 않습니다.

## 4. 거리형 게임의 전체 목표 지도

- 섬, 행성, 층, 다리처럼 멀리 가는 게임은 전체 목표 지도를 작게라도 계속 보여 줍니다.
- 지도는 현재 위치, 다음 목표, 마지막 목표를 한눈에 알게 해야 합니다.
- 지도는 보조 정보입니다. 큰 문제, 현재 단계, 선택지를 밀어내면 크기를 줄이거나 접습니다.
- 낮은 결과도 도착한 곳으로 보여 줍니다. 빈손이나 실패처럼 보이는 결과 화면은 쓰지 않습니다.

## 5. 보상과 확률

- 랜덤 보상은 유지합니다. 같은 계산을 해도 결과가 달라져야 다시 해 볼 이유가 생깁니다.
- 정답을 많이 맞히면 유리해야 하지만, 정답만으로 최고 보상이 쉽게 보장되면 안 됩니다.
- 최고 보상은 어렵게 둡니다. 한 판으로 자주 닿으면 재도전 동기가 약해집니다.
- 틀려도 무언가는 얻습니다. 낮은 결과에도 보이는 변화와 다시 할 길을 남깁니다.
- 한 차시의 중심 보상은 하나만 둡니다. 별도 코인, 랭킹, 수집장을 동시에 앞세우지 않습니다.

## 6. 보상 모달

- `보상 모달`은 크고 중앙에 둡니다. 학생이 방금 일어난 변화를 바로 보게 합니다.
- 버튼을 제외한 보이는 텍스트는 큰 결과 라벨 또는 변화량 1개로 줄입니다.
- 제목, 본문, 버튼이 같은 뜻을 반복하면 실패입니다. 버튼은 `다음`, `보기`, `다시`처럼 다음 행동만 말합니다.
- 이벤트 차이는 긴 설명보다 이미지, 아이콘, 변화량, 짧은 모션으로 구분합니다.
- 보상 모달에 매스몬이 보이면 생성 이미지 안에 처음부터 함께 들어가야 합니다. 배경을 먼저 만들고 기존 매스몬 PNG/WebP를 위에 얹는 합성은 쓰지 않습니다.
- 폴짝 뛰는 실제 이동이 꼭 필요하면 기존 캐릭터 컷아웃을 움직이지 말고, 생성형 스프라이트 프레임이나 짧은 영상처럼 그 모션 전용 자산을 따로 만듭니다.
- 효과가 문제, 선택지, 소리 버튼, 배지, 다음 조작부를 가리면 줄입니다.

## 7. 결과 화면과 도착 전 재기

- 결과 화면은 최종 점수보다 학생이 얻은 장소나 보상을 먼저 크게 보여 줍니다.
- 결과 공개 전에는 짧게 도착 전 재기, 길 스캔, 게이지 채우기 같은 기대 연출을 둘 수 있습니다.
- 결과 공개 뒤에는 한 줄 칭찬, 다시 하기 버튼, 간단한 요약만 남깁니다.
- 결과 배경은 생성형 bitmap 장면 1장을 기본으로 하고, 섬 이름·도착 라벨·칭찬 문구·다시 하기 버튼처럼 매 판 똑같은 요소는 이미지 안에 포함합니다.
- HTML/CSS 오버레이는 정답 수·점수처럼 매 판 달라지는 값만 맡깁니다.
- 결과 등급처럼 4단계 안에서만 바뀌는 라벨은 "매 판 달라지는 텍스트"가 아닙니다. 단계별 생성 이미지나 독립 타이틀/버튼 자산으로 바꾸고, HTML은 숨김 접근성 텍스트와 실제 점수만 남깁니다.
- 생성 이미지 위에 큰 반투명 카드, blur 패널, CSS 제목/본문, CSS로 그린 큰 다시 버튼을 올리면 실패입니다. 이 표준을 쓰는 차시는 `<main class="game">`에 `data-result-visual-standard="generated-assets"`를 붙여 공통 하네스 검사를 받습니다.
- 결과 화면을 통째 이미지로 요구받은 차시는 `data-result-render-mode="fullscene-score-slot"`를 함께 붙입니다. 이 모드의 보이는 HTML은 동적 점수 숫자 1개와 투명 다시하기 hitbox만 허용하며, 점수 박스 라벨은 이미지와 HTML 양쪽에서 보이지 않아야 합니다. 점수 중앙 정렬은 CSS 좌표 일치가 아니라 스크린샷 픽셀에서 숫자 글자 중심과 빈 점수칸 중심을 비교해 확인합니다.

## 8. 시각 자산 기준

- 첫 화면, 설명용 대표 장면, 보상 장면, 결과 장면은 생성형 bitmap 이미지를 기본 시각 자산으로 삼습니다.
- CSS와 HTML은 읽기 쉬운 문제, 선택지, 버튼, 배지, 오버레이, 상태 변화, 모션에 씁니다.
- 문제 풀이에 필요한 수식과 선택지는 이미지 안에 굽지 않습니다.
- 생성 이미지 위에 별도 캐릭터 파일을 붙여 장면을 만든 것처럼 보이게 하지 않습니다. 캐릭터가 장면의 일부여야 하면 프롬프트 단계에서 함께 생성합니다.
- 생성 원본 PNG와 배포용 WebP를 구분해 보관하고, 문서에 파일명과 오버레이 범위를 남깁니다.

## 9. 겹침/넘침 고정 규칙

- 버튼, 카드, 배지, 선택지 타일, 보상 모달, 결과 카드 안의 글자와 숫자는 상자 밖으로 나가면 실패입니다.
- 잘림, 삐져나옴, 줄 사이 겹침, 숫자나 기호가 칸 밖으로 밀리는 상태도 실패입니다.
- 문제, 선택지, 소리 버튼, 브랜드/단원 배지, 진행 상태, 보상 버튼, 다음 단계 조작부는 서로 덮으면 안 됩니다.
- 고정 포맷 UI에는 `aspect-ratio`, 명시적인 grid/flex track, `min/max-width`, `min/max-height`를 둡니다.
- 긴 문구는 먼저 학생 말로 짧게 줄입니다. 그래도 필요하면 `overflow-wrap`, 적절한 `line-height`, `max-width`, `fit-content`로 상자 안에서 자연스럽게 줄바꿈합니다.
- 간격과 컨테이너 크기에는 필요한 경우 `clamp()`를 쓰되, viewport-width 기반 글자 크기 축소로 억지로 맞추지 않습니다.
- hover, focus, 정답, 오답, disabled, 보상, 결과 상태에서도 선택지와 버튼 크기가 흔들리지 않아야 합니다.
