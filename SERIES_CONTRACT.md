# 에듀잇티 수학 게임 시리즈 계약

이 문서는 차시가 늘어나도 같은 시리즈처럼 보이도록 유지할 공통 약속입니다.
반복 제작 때 바로 대조할 실무 체크리스트는 루트 `LESSON_COMMONS.md`를 함께 봅니다.

## 학생 화면 공통 구조

### 첫 화면

- 왼쪽 위: `에듀잇티 수학 게임` 브랜드 배지
- 오른쪽 위: 학년/학기/단원 배지
- 왼쪽 아래: 배움주제 배지
- 중앙: 게임 제목(타이포그래픽 포스터/타이틀 아트)
- 제목 아래: 한 줄 목표
- 목표 아래: 생성형 버튼 아트로 만든 가장 큰 `시작` 버튼

첫 화면에는 긴 설명을 넣지 않습니다. 학생이 `시작`을 누르면 된다는 사실이 가장 먼저 보여야 합니다.

첫 화면 표준은 `data-cover-standard="generated-title-overlay"`입니다. `cover-generated.webp`는 글자 없는 대표 장면 배경으로 `.raster-bg`에 깔고, 게임명은 생성형 이미지로 만든 `.hero-title-art` 독립 오버레이로 얹습니다. 한 줄 목표는 짧은 HTML 텍스트로 둡니다. 새 차시와 생성형 시작 버튼으로 이관한 차시는 `data-cover-start-standard="generated-button-art"`를 함께 선언하고, 시작 버튼의 보이는 면을 CSS 텍스트 버튼이 아니라 생성형 이미지 버튼 자산(`start-button-generated.webp`)으로 둡니다. 실제 조작은 `<button class="cover-start-button" id="startButton" aria-label="시작"><img class="start-button-art" src="start-button-generated.webp" alt="" aria-hidden="true"></button>`처럼 같은 크기의 HTML 버튼이 맡습니다.

시작 버튼 아트는 1차시 포스터형 시작 버튼처럼 플레이 아이콘이 들어간 두툼한 노란 래스터 버튼을 기준으로 합니다. 1280×800 Stage 기준 너비 `400-460px`, 높이 `140-170px`, 비율 `2.6-3.0:1`을 기본으로 합니다. 배치는 제목/목표 묶음의 시선 흐름을 따르며, 보통 목표 아래 `14-24px` 간격, Stage y좌표 `500-580px` 안쪽에 둡니다. 1차시의 물성 있는 버튼감과 2·3차시의 제목-목표-버튼 흐름을 기준으로 삼고, 배경 주인공을 가리지 않는 쪽을 우선합니다.

`cover-generated.webp` 한 장에 제목·목표·시작 버튼을 모두 구워 넣거나, 커버 전체를 투명 hitbox로 시작시키는 방식은 새 차시에서 금지합니다. 버튼 크기의 실제 HTML 버튼은 접근성과 클릭을 위해 필요하지만, 보이는 버튼 표면을 CSS 배경/텍스트로 새로 그리면 실패입니다. 아직 이전 커버 구조를 쓰는 차시는 `<main class="game">`에 `data-cover-standard="legacy-raster-poster"`를 붙여 예외임을 드러내고, 복제 기준으로 삼지 않습니다.

기존 `generated-title-overlay` 차시 중 `.primary-button` 시작 구조는 개별 이관 전까지 `compatibility-primary-button`으로 분류합니다. 이 상태는 기존 차시 실행 호환을 위한 것으로, 새 차시나 새로 이관하는 커버의 목표 표준은 생성형 버튼 아트입니다.

첫 화면 제목은 단순히 큰 HTML 글자를 올리는 것으로 끝내지 않습니다. 사용자가 `그림으로`, `GPT Image`, `제목 이미지`를 요구하면 기존 커버 배경은 유지하고, 제목 부분만 별도 래스터 타이틀 아트로 생성해 얹습니다. 전체 커버 이미지를 새로 갈아엎거나 HTML/CSS/SVG로 제목 이미지를 흉내 내지 않습니다. 접근성용 실제 제목은 숨김 텍스트로 남깁니다.

생성형 이미지로 제목을 만들 때는 `독립 제목 로고`, `정확한 한글`, `전체 장면 아님`, `초록 배경 또는 투명 배경` 기준으로 만들고, 배경 제거 후 첫 화면 위에 올립니다. 시작 버튼도 `standalone poster-style start button asset`, `exact Korean text 시작`, `play icon`, `glossy yellow pill`, `transparent or chroma-key background` 기준으로 생성합니다. 현재 표준은 1차시 생성형 커버의 시작 버튼 물성을 따른 독립 자산입니다. 한글 철자, 제목 크기, 목표 문구·시작 버튼 아트·하단 배지와의 간격은 실제 캡처로 검수합니다. 철자 오류, 어색한 이미지, 화면과 맞지 않는 제목/버튼 자산은 첫 화면에 남기지 않습니다.

보상 캐릭터나 대표 이미지는 결과 화면에서 강하게 보여 줍니다. 첫 화면에 넣을 때는 아주 절제해서 쓰고, 구조가 흐려지면 빼는 쪽을 우선합니다.

배움주제 배지는 단원 배지와 같은 형태를 쓰되 주황색 계열로 구분합니다. 문구는 차시의 실제 학습 내용을 짧게 적습니다.

### 설명 화면

- 2-3개의 짧은 단계만 보여 줍니다.
- 버튼은 다음 행동을 직접 말합니다. 예: `문제 시작`, `탐험 시작`, `고르기 시작`
- 설명 화면은 매뉴얼이 아니라 첫 문제로 가기 전 작은 예열입니다.

### 문제 화면

- 문제와 선택지가 가장 큽니다.
- 점수, 콤보, 진행도는 보조 정보입니다.
- 보상 상자는 문제를 맞힌 뒤에만 주인공이 됩니다.
- 문제 화면은 "3초 안에 무엇을 고를지 알 수 있는가?"를 통과해야 합니다.
- 기본 화면에는 `큰 문제`, `현재 단계 조작판`, `한 줄 지시문`, `선택지`만 둡니다. 개념 설명판, 풀이 해설판, 힌트판, 계산 미리보기, 보상 상태판을 동시에 펼치지 않습니다.
- 단계형 문제는 현재 단계만 크게 보여 주고, 이전 단계는 작은 완료 칩으로 접으며, 다음 단계는 `?` 또는 잠금 상태로 둡니다.
- 수학 원리는 긴 문장 대신 블록, 칸, 스티커, 화살표, 자리값 묶음처럼 조작 가능한 시각 구조로 보여 줍니다.
- 힌트는 기본으로 닫아 둡니다. 열었을 때도 지금 단계 힌트 1개만 보여 주고 전체 풀이 과정을 한 번에 펼치지 않습니다.
- 문제 화면에 문장형 해설을 여러 줄 쓰지 않습니다. 한 줄 지시문으로 줄일 수 없으면 화면 설계를 다시 합니다.

### 보상 화면

- 상자는 중앙 또는 문제 카드보다 시선이 잘 가는 위치에 둡니다.
- 보상 결과는 큰 글자와 점수 변화로 보여 줍니다.
- 보상은 한 번만 적용됩니다.
- 보상/피드백 모달은 버튼을 제외하고 보이는 텍스트를 한 덩어리로 제한합니다. 기본은 큰 결과 라벨 또는 변화량 1개이며, 긴 설명 문장은 화면에 기본 노출하지 않습니다.
- 모달 안에서 같은 뜻의 말을 반복하지 않습니다. `출하!`와 `출하 보기`, `결과`와 `결과 보기`처럼 제목·본문·버튼이 같은 말을 되풀이하면 실패입니다.
- 보상 모달에 매스몬이 필요하면 생성 이미지 안에 처음부터 같이 들어가야 합니다. 배경 이미지 위에 기존 매스몬 PNG/WebP를 따로 올리는 임시 합성은 금지합니다.
- 실제로 뛰거나 날아가는 모션을 보여 줘야 할 때는 생성형 스프라이트/프레임 같은 모션 전용 자산을 만들고, 기존 캐릭터 컷아웃을 CSS로 얹어 움직이는 방식은 쓰지 않습니다.

### 결과 화면

- 학생이 얻은 장소나 보상 장면을 먼저 크게 보여 줍니다.
- 섬 이름·도착 라벨·칭찬 문구·다시하기 버튼처럼 매 판 똑같은 요소는 생성 이미지 안에 포함하는 것을 기본값으로 합니다.
- HTML/CSS 오버레이는 정답 수·점수처럼 매 판 달라지는 값만 맡깁니다.
- 결과 등급처럼 유한 단계로만 바뀌는 라벨·칭찬·버튼 장식은 단계별 생성 이미지나 독립 생성형 타이틀/버튼 자산으로 처리합니다. 보이는 CSS 결과 카드, 반투명 blur 패널, CSS 제목/본문, CSS로 그린 큰 다시 버튼은 실패입니다.
- 생성형 결과 라벨/버튼 자산 표준을 쓰는 차시는 `<main class="game">`에 `data-result-visual-standard="generated-assets"`를 선언해 공통 하네스가 검사할 수 있게 합니다.
- 결과 화면을 통째 생성 이미지로 쓰는 차시는 `data-result-render-mode="fullscene-score-slot"`를 함께 선언합니다. 이 모드에서 학생 눈에 보이는 HTML은 동적 점수 숫자 1개와 투명 hitbox뿐이며, 점수 박스 라벨은 생성 이미지와 HTML 어디에도 보이면 안 됩니다. 점수 숫자가 빈칸 중앙에 있는지는 CSS 좌표값 통과가 아니라 실제 스크린샷 픽셀 중심 비교로 확인합니다.
- 하단 요약이나 별도 정보 박스는 꼭 필요할 때만 1개 이하로 둡니다.

## 보상 구조 원칙

- 한 차시의 중심 보상은 하나만 둡니다.
- `점수`, `상자`, `최종 캐릭터`처럼 연결되는 구조는 허용합니다.
- 별, 하트, 배지, 코인, 캐릭터, 랭킹을 동시에 전면 배치하지 않습니다.
- 확률 보상은 기대감과 긴장감이 같이 있어야 합니다.

## 문구 원칙

- 학생이 읽는 문장은 짧고 자연스럽게 씁니다.
- 버튼은 행동만 남깁니다. 예: `시작`, `열기`, `다음`, `다시`
- 내부 용어는 화면에 쓰지 않습니다.
- 학생 화면에는 `AI Mart`라는 이름을 쓰지 않습니다.
- 문제 화면의 지시문은 한 줄을 기본으로 합니다. 예: `먼저 63×3을 골라요`, `7과 곱한 값을 골라요`.
- 학생 화면에 `핵심 메커니즘`, `메타 필드`, `부분곱`, `정답 수 게이트` 같은 제작자 용어를 쓰지 않습니다.
- 모달 버튼은 다음 행동만 말하고, 위의 결과 라벨과 같은 단어를 반복하지 않습니다. 결과 라벨이 `바로 결과`라면 버튼은 `보기`처럼 줄입니다.

## UI 넘침·겹침 금지

- 학생에게 보이는 텍스트가 버튼, 카드, 배지, 선택지, 보상 모달, 결과 카드 밖으로 튀어나가면 실패입니다. 글자 잘림, 줄 사이 겹침, 숫자·기호가 칸 밖으로 나가는 상태도 실패입니다.
- 문제, 선택지, 소리 버튼, 브랜드/단원 배지, 진행 상태, 보상 버튼, 다음 단계 조작부가 서로 덮어서 읽기나 클릭을 방해하면 실패입니다.
- 고정 포맷 UI는 `aspect-ratio`, 명시적인 grid/flex track, `min/max-width`, `min/max-height`, 예약 공간으로 크기를 안정시킵니다. hover, 정답/오답, disabled, 긴 숫자, 긴 문구 때문에 레이아웃이 흔들리면 다시 설계합니다.
- 긴 문구는 먼저 줄입니다. 필요한 경우 상자 안에서 줄바꿈되게 하되, 상자 밖으로 밀어내거나 viewport 폭 기준 글자 크기 조절로 억지로 맞추지 않습니다.
- 화면을 완성했다고 말하기 전에 컴퓨터 화면과 태블릿 가로 화면에서 첫 화면, 설명, 문제, 보상 모달, 결과를 캡처해 텍스트 넘침과 요소 겹침이 없는지 확인합니다.
- 각 차시 `REPORT.md`에는 "텍스트 넘침·요소 겹침 QA"를 적고, 확인한 화면 크기와 화면 상태를 남깁니다.

## 기기 기준

- 기준 화면: 태블릿 가로, 컴퓨터
- 스마트폰 세로 화면은 기본 지원 대상이 아닙니다.
- 긴 숫자, 긴 한글 이름, 큰 보상 텍스트가 칸을 넘지 않는지 확인합니다.

## Stage 비율 계약

- 모든 차시는 기본 Stage 비율을 `16:10`으로 통일합니다.
- 기준 제작 크기는 `1280×800`입니다. 첫 화면·설명·문제·보상·결과 화면은 이 비율 안에서 설계하고, 스크린샷도 이 비율을 우선합니다.
- PC와 태블릿 가로 화면에서는 Stage를 `contain` 방식으로 맞춥니다. 남는 영역은 바깥 배경 여백으로 처리하고, Stage 내부 UI를 늘려 비율을 깨지 않습니다.
- 모든 `index.html`의 `<main class="game">`에는 `data-stage-ratio="16:10"`과 `data-stage-size="1280x800"`을 둡니다.
- 모든 `index.html`은 `.stage-shell`이 `width: min(1280px, calc((100dvh - 48px) * 1.6), 100%);`, `aspect-ratio: 16 / 10;`을 담당하고, `.screen`은 그 안을 `position: absolute; inset: 0; width: 100%; height: 100%;`로 채웁니다.
- `소리` 같은 전역 조작 버튼은 Stage 밖 viewport에 fixed로 띄우지 않습니다. `.stage-shell` 안의 상단 오른쪽 보조 슬롯에 작게 두고, `top-row`/`hud`는 그 공간만큼 비워 배지·문제·선택지를 가리지 않게 합니다.
- 차시를 만들거나 화면을 크게 바꾼 뒤에는 루트에서 `node scripts/check-stage-ratio.mjs`를 실행해 Stage 계약을 통과해야 합니다.

- `manifest.json`에 차시가 추가되었는가?
