## UI 텍스트 넘침·요소 겹침 금지 (강제)

앞으로 모든 차시의 화면은 "글자가 자기 상자 안에 들어가 있는가?", "요소끼리 서로 덮지 않는가?"를 통과해야 합니다. 통과하지 못하면 완성 화면이 아니라 실패 화면으로 보고 레이아웃이나 문구를 다시 줄입니다.

- 버튼, 카드, 배지, 선택지, 보상 모달, 결과 카드 안의 한글·숫자·기호는 상자 밖으로 튀어나가면 실패입니다. 잘림, 삐져나옴, 줄 사이 겹침도 실패입니다.
- 문제, 선택지, 설정/소리 버튼, 브랜드/단원 배지, 진행 상태, 보상 버튼, 다음 단계 조작부가 서로 덮이면 실패입니다. 의도한 오버레이가 아니라 읽기나 클릭을 방해하면 무조건 다시 배치합니다.
- 고정 포맷 UI(지도, 보드, 선택지 그리드, 배지 줄, 버튼 줄, 결과 카드)는 `aspect-ratio`, 명시적인 grid/flex track, `min/max-width`, `min/max-height`, 예약 공간(`--sound-reserve` 등)으로 크기를 안정시킵니다. hover, 정답/오답, disabled, 긴 숫자, 긴 문구 때문에 카드나 버튼 크기가 흔들리면 실패입니다.
- 긴 문구는 CSS로 억지로 밀어 넣지 말고 먼저 학생 말로 짧게 줄입니다. 그래도 길면 상자 안에서 자연스럽게 줄바꿈되게 하고, `overflow-wrap`, 적절한 `line-height`, `max-width`를 둡니다.
- 글자를 맞추려고 viewport 폭 기준으로 글자 크기를 계속 줄이는 방식은 쓰지 않습니다. 글자 크기는 화면 목적에 맞게 정하고, 공간이 부족하면 레이아웃·문구·정보량을 줄입니다.
- 스크린샷 QA는 컴퓨터 화면과 태블릿 가로 화면을 모두 봅니다. 첫 화면, 설명, 문제 1단계, 문제 2단계, 보상 모달, 결과 화면에서 텍스트 넘침·요소 겹침·클릭 영역 충돌이 0개여야 합니다.
- `REPORT.md`에는 "텍스트 넘침·요소 겹침 QA"를 남기고, 어떤 화면 크기와 어떤 화면 상태를 확인했는지 적습니다.

## 정밀 동적 UI 화면 패턴 (전국 순위판 등)

전국 순위판, 대시보드, 표, 여러 행 목록처럼 매 판 달라지는 값이 많고 정확한 정렬이 필요한 화면은 생성 이미지 안 박스에 글자를 맞추지 않습니다.

- 생성 이미지는 축하 배경, 무대, 불꽃, 매스몬 장식, 분위기만 맡깁니다. 프롬프트에는 `no text`, `no letters`, `no numbers`, `no UI panels`, `no leaderboard board`, `no cards`, `no table rows`, `no buttons`, `no empty boxes`, `no labels`를 명시합니다.
- 순위판, 내 기록 박스, 행 배경, 스크롤 영역, 버튼 표면, 보이는 글자는 하나의 SVG `viewBox="0 0 1280 800"` 컴포넌트가 그립니다. 배경 이미지와 UI를 같은 SVG 좌표계 안에 두어 Stage가 줄어도 함께 줄어들게 합니다.
- 실제 클릭은 같은 좌표의 투명 HTML `button` hitbox가 맡고, 버튼 안에는 보이는 HTML 텍스트를 넣지 않습니다. 접근성용 `aria-label`만 둡니다.
- 긴 이름과 상태 문구는 렌더러에서 정해진 글자 수로 말줄임합니다. 글자를 맞추려고 viewport 폭 기준 글자 크기를 계속 줄이지 않습니다.
- 목록은 고정 row 슬롯 안에서 안정적으로 보여 주고, 10위까지 같은 SVG 리스트 안에서 wheel/pointer 스크롤로 확인하게 합니다.
- 이 패턴은 동적 UI가 많은 화면을 위한 예외입니다. 결과 라벨, 칭찬, 고정 버튼 장식처럼 유한하고 고정된 시각 요소는 여전히 생성형 결과 이미지/타이틀/버튼 자산으로 처리합니다.
- QA는 1280×800, 1024×768, 사용자가 올린 문제 크기와 비슷한 브라우저 크기에서 캡처하고, SVG `<text>`의 `getBBox()`가 Stage와 의도한 보드 영역 밖으로 나가지 않는지 확인합니다.

## 초3 문제 화면 과밀 금지 (강제)

앞으로 모든 차시의 문제 화면은 "초등학교 3학년 학생이 3초 안에 지금 무엇을 해야 하는지 알 수 있는가?"를 통과해야 합니다. 통과하지 못하면 완성 화면이 아니라 실패 화면으로 보고 다시 줄입니다.

- 문제 화면의 목적은 **현재 한 단계의 수학 행동**을 하게 하는 것입니다. 개념 설명판, 풀이 해설판, 힌트판, 계산 미리보기, 보상 상태판을 한 화면에 동시에 펼치지 않습니다.
- 한 문제 화면에 기본으로 보여도 되는 학습 요소는 4개뿐입니다: **큰 문제**, **현재 단계 조작판**, **한 줄 지시문**, **선택지**. 나머지는 접거나, 완료 뒤에 작은 체크칩으로 줄이거나, 힌트 버튼 뒤로 숨깁니다.
- 설명 문구는 기본 **한 줄**입니다. 문제 화면에서 문장형 해설을 여러 줄 쓰지 않습니다. 예: `먼저 63×3을 골라요`, `7과 곱한 값을 골라요`. 교사용 설명처럼 긴 문장은 금지합니다.
- 원리는 글로 설명하지 말고 **보이는 조작**으로 드러냅니다. 자리값, 0 붙이기, 내림, 나머지, 검산은 블록·칸·스티커·화살표처럼 학생이 바로 볼 수 있는 구조로 보여 줍니다.
- 단계형 문제는 **현재 단계만 크게** 보여 줍니다. 이전 단계는 작은 완료 칩으로 접고, 다음 단계는 `?` 또는 잠금 상태로 둡니다. 여러 단계의 계산판을 옆에 동시에 펼치지 않습니다.
- 단계 정답 뒤에는 반드시 **정답 확인 상태**가 보입니다. 학생이 방금 고른 답이 어디에 들어갔는지 계산판·칸·블록에서 바로 확인하고, 그 확인 화면이 잠깐 머문 뒤 다음 단계로 넘어가야 합니다. 마지막 단계 뒤 보상은 자동 모달보다 학생이 확인 버튼을 눌러 여는 흐름을 우선합니다.
- 선택지는 항상 가장 눈에 잘 띄는 한 덩어리로 둡니다. 선택지와 큰 문제 사이에 장식, 보상판, 긴 설명 패널을 끼워 넣지 않습니다.
- 문제·현재 단계·선택지가 Stage의 대부분을 차지해야 합니다. 보조 정보(진행도, 콤보, 보상 상태, 설정/소리 버튼)는 작게 두고, 풀이 판단을 방해하면 숨깁니다.
- 힌트는 기본으로 열어 두지 않습니다. 학생이 눌렀을 때도 **지금 단계 힌트 1개**만 보여 주고, 전체 풀이 과정을 한 번에 펼치지 않습니다.
- 학생 화면에 `핵심 메커니즘`, `메타 필드`, `부분곱`, `정답 수 게이트` 같은 제작자 용어를 쓰지 않습니다. 수학 용어가 필요하면 교과서식 짧은 말로 바꿉니다.
- 스크린샷을 보고 "문제보다 패널이 많다", "읽어야 할 문장이 많다", "지금 눌러야 할 것이 바로 안 보인다"는 느낌이 들면 무조건 덜어냅니다.

## 핵심 설계 철학 (모든 차시 공통)

이 작업실이 만드는 모든 게임은 아래 철학을 따릅니다. 이것은 차시별 소재가 바뀌어도 변하지 않는, 이 자료의 정체성입니다. 새 차시를 만들 때마다 문제 생성·보상 설계·결과 구조가 이 철학에 맞는지 먼저 확인합니다.

### 1. 계산은 "행동의 도구"다

문제는 그 자체로 끝나는 반복 과제가 아니라, 점수·등급·결과 보상으로 이어지는 행동의 도구입니다. 학생은 문제를 맞혀야 다음 보상(상자·연료·부품·바구니 등 차시 소재)을 열 수 있고, 그 결과에 따라 보상이 크게 오르내립니다. 계산은 예측 가능한 영역이지만 보상은 예측 불가능하게 설계해, 학생은 같은 유형의 문제를 풀면서도 매번 다른 결과를 경험합니다.

교육적으로는 반복 연습에 게임의 간헐적 보상을 결합한 구조입니다. 학생은 정답을 맞히는 데서 멈추지 않고, 다음 보상과 최종 결과를 기대하며 더 많은 문제에 자발적으로 접근합니다. 이때 무작위성은 학습을 흐리는 장치가 아니라, 이미 배운 계산 절차를 더 자주 사용하게 만드는 동기 장치로 작동합니다.

### 2. 무작위성은 반드시 유지한다

랜덤 보상은 빼지 않습니다. 아이들이 흥미를 가지고 다시 도전하게 만드는 핵심 동력입니다. 단, 무작위성은 "계산을 안 해도 되는 핑계"가 아니라 "이미 한 계산의 결과를 매번 새롭게 만드는 장치"여야 합니다. 보상에는 기대감과 긴장감이 함께 있어야 합니다(잘 풀어도 가끔 손해, 운이 따르면 한 번에 도약).

### 3. 정답이 전부를 보장하지 않는다

- 좋은 것은 얻기 어렵게 만듭니다. 한 판으로 최고 등급에 쉽게 닿으면 반복할 이유가 사라집니다.
- 정답을 많이 맞히는 것은 분명히 유리해야 합니다. 하지만 그것만이 모든 걸 쉽게 얻는 유일한 통로가 되어서는 안 됩니다.
- 못하는 아이도 무언가는 반드시 얻습니다(빈손 금지). 동시에 운이 따르면 평소보다 더 좋은 것을 얻을 수도 있습니다.
- 그래서 잘하는 아이는 실력으로, 아직 서툰 아이는 실력+운으로 위를 노립니다. 누구도 완전히 막히지 않고, 누구도 한 번에 다 갖지 않습니다.

### 4. "다음엔 더 멀리"를 만든다

결과는 차시마다 자체 완결형 등급으로 보여 줍니다. 학생이 "이번엔 여기까지 왔지만 다음엔 더 멀리 갈 수 있다"고 느끼게 만드는 것이 이 자료의 핵심입니다. 한 개의 콘텐츠를 여러 번 반복하게 만드는 재도전 동기가 모든 차시 설계의 최종 점검 기준입니다.

### 차시 설계 점검 질문

새 차시를 만들 때 아래에 모두 "그렇다"가 나와야 합니다.

- 문제를 푸는 것이 보상으로 이어지는 "행동의 도구"로 설계되었는가?
- 랜덤 보상이 있고, 기대감과 긴장감이 함께 있는가?
- 최고 보상은 충분히 얻기 어려운가? (한 판으로 쉽게 닿지 않는가)
- 정답이 유리하되, 정답만이 모든 걸 쉽게 주는 유일한 통로는 아닌가?
- 못해도 빈손은 아니고, 운으로 한 단계 더 갈 여지가 있는가?
- 학생이 "다음엔 더 멀리"를 떠올리며 다시 하고 싶어지는가?

이 작업실은 한 차시짜리 게임이 아니라 "한 시리즈"를 만듭니다. 차시가 24개로 늘어도 학생이 보기에 같은 브랜드의 게임으로 느껴져야 합니다.

### 시리즈 정체성

- 시리즈 통합 마스코트는 `매스몬`입니다. 매스몬은 첫 화면 동행 캐릭터·브랜드 정체성으로 쓰고, 결과에서는 차시별 중심 보상 등급 뒤에 이번 판 보상 카드 1장으로 보여 줍니다(저장형 도감처럼 모아두는 구조는 만들지 않습니다 — 별도 백엔드 없음).

- 화면 골격(첫 화면 → 설명 → 문제 → 보상 → 결과)은 모든 차시가 동일하게 따릅니다. 자세한 규칙은 `SERIES_CONTRACT.md`가 단일 기준입니다.

- 한 차시는 기본적으로 한 매스몬 팩만 씁니다. 여러 팩을 섞어야 할 때는 해당 차시 `README.md`/`REPORT.md`와 팩 `manifest.json`에 이유를 남깁니다.
- 새 팩을 만들기 전 기존 팩의 contact sheet와 manifest를 보고 실루엣·동물종·소재가 겹치지 않는지 확인합니다.
- 이미지 분위기는 팩 단위로 반드시 일관되어야 합니다. 앞으로 새로 만드는 매스몬은 1차시 `매스몬 상자런`의 밝은 2D 애니/스티커형 톤(`mathmon-v1-anime-sticker`)을 기본값으로 삼습니다. 현재 실행 기준은 `base-pack`과 `zero-factory-animal-pack`이며, 0 공장 기존 팩과 V2 팩은 보존만 합니다.
- 차시별 전용 매스몬도 본체는 동물/판타지 생물이어야 합니다. 차시 테마는 소품·의상·배지·포즈로만 표현하고, 톱니바퀴·자석·상자·컨베이어 같은 사물 자체를 매스몬 몸으로 만들지 않습니다.

### RasterStage 화면 규칙

- 모든 새 차시의 첫 화면과 마지막 결과 화면은 이미지 생성으로 만든 대표 장면을 배경으로 사용합니다.
- 모든 RasterStage와 주요 게임 화면은 `1280×800` 기준 `16:10` 안에서 설계합니다. 생성 이미지가 다른 비율이어도 Stage 안에서는 `object-fit: cover/contain`과 HTML 오버레이로 맞추고, Stage 자체 비율을 바꾸지 않습니다.
- 첫 화면 배경 이미지는 글자 없는 대표 장면이어야 합니다. 게임명·한 줄 목표·시작 버튼을 한 이미지에 구워 넣지 않고, `cover-generated.webp` 배경 + `.hero-title-art` 제목 이미지 + HTML 목표 + 생성형 시작 버튼 아트로 나눕니다.
- 현재 목표 표준은 실제 HTML 버튼 안에 생성형 버튼 아트를 넣는 구조입니다. 새 차시와 이관 완료 차시는 `data-cover-start-standard="generated-button-art"`로 선언합니다. 기존 `.primary-button` 시작 커버는 `compatibility-primary-button`으로 분류해 개별 이관 전까지 유지할 수 있지만, 보이는 CSS 텍스트 버튼과 커버 전체 투명 hitbox 방식은 새 차시에 쓰지 않습니다.
- 첫 화면 제목은 배경 이미지 위에 별도의 타이틀 아트로 얹을 수 있습니다. 이때 **생성 대상은 전체 커버가 아니라 제목 로고만**입니다. 기존 `cover-generated.webp` 같은 대표 장면을 유지하고, GPT Image 등으로 만든 제목 로고를 초록 배경 제거 또는 투명 PNG/WebP로 분리해 `.hero-title-art`로 올립니다.
- 학생에게 보이는 제목은 하나의 그림처럼 보여야 합니다. HTML/CSS로 급조한 밋밋한 큰 글자, 어설픈 SVG 로고, 전체 커버를 새로 합성해 장면 구도를 망가뜨리는 방식, AI가 철자를 틀린 한글 이미지는 모두 실패로 봅니다.
- 제목 로고를 만드는 순간에는 `image_gen`/GPT Image 등 생성형 이미지 도구를 실제로 사용합니다. 로컬 폰트로 직접 래스터화하거나 CSS 효과를 캡처해 만든 파일은 `.hero-title-art`에 연결하지 않습니다.
- 생성형 이미지처럼 보여야 하는 화면·타이틀·보상·결과 자산을 로컬 폰트, Pillow, canvas, SVG, CSS 캡처, 기존 PNG/WebP 겹치기 등 로컬 합성으로 만들지 않습니다. 로컬 합성은 사용자가 먼저 명시적으로 허락한 경우에만 예외로 쓸 수 있습니다.
- 배경 제거, 크롭, 리사이즈, WebP 변환, 용량 최적화처럼 생성형 원본의 의미를 바꾸지 않는 후처리는 허용합니다. 새 문구·버튼·캐릭터·패널을 로컬에서 그려 붙이면 후처리가 아니라 금지된 로컬 합성입니다.
- GPT Image 등 생성형 이미지로 제목을 만들 때는 `standalone title/logo asset`, `exact Korean text`, `no full scene`, `flat chroma-key background` 조건으로 만들고, 한글 철자를 화면 캡처로 검수합니다. 철자가 틀리거나 품질이 맞지 않으면 같은 제목 오버레이 슬롯에서 재생성합니다.
- 결과 화면 생성 이미지는 차시의 중심 보상 장면과 매스몬 동행 캐릭터가 자연스럽게 들어간 한 장면으로 만듭니다. 섬 이름·도착 라벨·칭찬 문구·다시하기 버튼처럼 매 판 똑같은 요소는 개별 HTML/CSS 조각으로 다시 만들지 말고 생성 이미지 안에 포함하는 것을 기본값으로 합니다.
- 결과 화면에서 HTML/CSS 오버레이로 남길 것은 정답 수·점수처럼 매 판 실제로 달라지는 값뿐입니다. 버튼 시각 요소를 이미지 안에 넣은 경우에도 키보드·스크린리더 접근성을 위해 같은 위치의 실제 HTML 버튼 또는 hitbox는 유지할 수 있습니다.
- 결과 화면에 보이는 큰 결과 라벨, 칭찬 문구, 다시 버튼 장식은 CSS 텍스트/카드로 만들지 않습니다. 생성 이미지 위에 반투명 카드, blur 패널, CSS 제목, CSS 본문, CSS로 그린 큰 버튼을 얹으면 하네스 실패입니다.
- 결과 라벨이 4단계처럼 유한하게 바뀌면 각 단계별 생성형 결과 이미지나 독립 타이틀/버튼 자산을 만들고 JS로 이미지 `src`만 바꿉니다. HTML 텍스트는 `visually-hidden` 접근성 값이나 실제 정답 수·점수처럼 매 판 계산되는 최소 정보만 남깁니다.
- 생성형 결과 라벨/버튼 자산 표준을 쓰는 차시는 `<main class="game">`에 `data-result-visual-standard="generated-assets"`를 선언해 공통 하네스가 CSS 결과 카드와 보이는 HTML 결과 문구를 잡게 합니다.
- 결과 화면을 통째 이미지로 요구받거나 `result-final-*-generated.webp`가 도착 상태별 완성 장면일 때는 `<main class="game">`에 `data-result-render-mode="fullscene-score-slot"`도 함께 선언합니다. 이 모드에서 보이는 HTML은 동적 점수 숫자 1개(`#finalCorrectText`)와 투명 다시하기 hitbox뿐입니다.
- `fullscene-score-slot`의 점수 숫자는 이미지 안 빈 점수칸에 맞춘 RasterStage 슬롯으로 배치합니다. 생성 이미지마다 빈칸 위치가 다르면 `data-result-island` 같은 상태별 슬롯을 명시합니다. 중앙 정렬 QA는 CSS 좌표값만 보지 말고, 1280×800과 1024×768 스크린샷 픽셀에서 숫자 글자 중심과 빈 점수칸 중심을 비교해 확인합니다.
- `fullscene-score-slot` 모드에서는 `.result-stats`, `.result-stat`, `.result-card`, `.result-copy` 같은 CSS 결과 카드와 보이는 `resultTitle`/본문/버튼 장식이 있으면 실패입니다. 점수 박스 라벨도 이미지와 HTML 양쪽에서 보이지 않게 합니다.
- 사용자가 "이미지에는 글자를 넣지 말라"는 계획을 줬더라도 결과 화면의 고정 라벨·칭찬·버튼을 CSS 카드로 대체하지 않습니다. 둘이 충돌하면 구현 전에 계획을 바로잡고, 글자 없는 결과를 택한다면 보이는 CSS 라벨도 함께 제거합니다.
- 결과 화면에 멋진 보상 장면과 정확한 동적 UI가 함께 필요하면 `data-result-render-mode="hybrid-generated-dynamic"`을 씁니다. 생성 이미지는 로봇·섬·행성·무대·빛·감정 같은 결과 세계를 맡고, SVG `viewBox="0 0 1280 800"` 오버레이는 정답 수, 이번 판 결과명, 짧은 진행값, 버튼 표면처럼 정확한 정렬이 필요한 값만 맡습니다.
- 혼합형 결과 화면의 생성 프롬프트에는 `no text`, `no letters`, `no numbers`, `no score board`, `no stats cards`, `no table`, `no buttons`, `no UI panels`를 기본으로 넣습니다. 이미지 안에는 텍스트 없는 받침대, 홀로그램 빔, 빈 에너지 장치 같은 장면 소품만 허용합니다.
- 혼합형 SVG 오버레이는 CSS 카드 우회가 아닙니다. 보이는 SVG 텍스트는 1-3개 짧은 값으로 제한하고, 긴 칭찬·고정 결과 라벨·버튼 장식은 생성형 이미지나 독립 생성형 타이틀/버튼 자산으로 처리합니다. 동적 값이 4개 이상 필요하면 결과 화면이 아니라 순위판/대시보드 패턴으로 분리합니다.
- 보상 모달이나 결과 장면에서 매스몬이 필요하면 이미지 생성 단계에서 장면 안에 매스몬을 함께 포함합니다. 완성된 배경 위에 기존 매스몬 PNG/WebP를 따로 붙여 캐릭터처럼 보이게 합성하는 방식은 어색한 결과가 나기 쉬우므로 쓰지 않습니다. 실제 이동 애니메이션이 꼭 필요할 때만 별도의 생성형 스프라이트/프레임을 만들고, 기존 캐릭터 컷아웃을 배경 위에 올리는 임시 합성은 금지합니다.
- 생성 원본 PNG는 작업실에 보관하고, 학생에게 배포하는 `index.html`에서는 WebP 같은 경량 포맷을 우선 사용합니다.
- 문제 화면과 선택지 화면은 생성 이미지보다 읽기 쉬운 HTML/CSS UI를 우선합니다. 장면 이미지는 문제와 선택지를 가리지 않는 배경 또는 보상 연출에만 씁니다.
- `README.md`와 `REPORT.md`에는 사용한 생성 이미지 파일명, 배치 의도, 동적 HTML 오버레이 범위를 기록합니다.

## 배포 (GitHub Pages)

학생용 정적 HTML 게임은 이 레포지토리 하나의 GitHub Pages로 공유합니다(별도 공개 레포 금지). 필요한 게임 폴더만 `actions/upload-pages-artifact`와 `actions/deploy-pages`로 배포하고, 공유 전 `curl -I -L <url>`로 실제 URL이 `HTTP 200`인지 확인합니다. Pages 실패 시 `gh run view <run-id> --log-failed`로 원인을 먼저 확인합니다. checkout 단계에서 submodule 오류가 나면 `.gitmodules`와 `git ls-files -s`로 깨진 `160000` gitlink를 확인하고, 필요 없는 임시 gitlink는 `git rm <path>`로 제거합니다.
