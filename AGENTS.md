## 핵심 설계 철학 (모든 차시 공통)

이 작업실이 만드는 모든 게임은 아래 철학을 따릅니다. 차시별 소재가 바뀌어도 변하지 않는 기준이며, 새 차시를 만들 때마다 문제 생성·보상 설계·결과 구조가 이 철학에 맞는지 먼저 확인합니다.

### 1. 계산은 "행동의 도구"다

문제는 그 자체로 끝나는 반복 과제가 아니라, 점수·등급·결과 보상으로 이어지는 행동의 도구입니다. 학생은 문제를 맞혀야 다음 보상(상자·연료·부품·바구니 등 차시 소재)을 열 수 있고, 그 결과에 따라 보상이 크게 오르내립니다.

계산은 예측 가능한 영역이지만 보상은 예측 불가능하게 설계합니다. 학생은 같은 유형의 문제를 풀면서도 매번 다른 결과를 경험하고, 최종 보상을 기대하며 같은 계산 절차를 반복하게 됩니다.

### 2. 무작위성은 반드시 유지한다

랜덤 보상은 빼지 않습니다. 아이들이 흥미를 가지고 다시 도전하게 만드는 핵심 동력입니다. 단, 무작위성은 "계산을 안 해도 되는 핑계"가 아니라 "이미 한 계산의 결과를 매번 새롭게 만드는 장치"여야 합니다.

보상에는 기대감과 긴장감이 함께 있어야 합니다. 잘 풀어도 가끔 손해가 날 수 있고, 운이 따르면 한 번에 크게 도약할 수 있습니다.

### 3. 정답이 전부를 보장하지 않는다

- 좋은 것은 얻기 어렵게 만듭니다. 한 판으로 최고 등급에 쉽게 닿으면 반복할 이유가 사라집니다.
- 정답을 많이 맞히는 것은 분명히 유리해야 합니다. 하지만 정답만이 모든 걸 쉽게 얻는 유일한 통로가 되어서는 안 됩니다.
- 못하는 아이도 무언가는 반드시 얻습니다. 빈손으로 끝나는 판은 만들지 않습니다.
- 잘하는 아이는 실력으로, 아직 서툰 아이는 실력+운으로 위를 노립니다. 누구도 완전히 막히지 않고, 누구도 한 번에 다 갖지 않습니다.

### 4. "다음엔 더 멀리"를 만든다

결과는 차시마다 자체 완결형 등급으로 보여 줍니다. 학생이 "이번엔 여기까지 왔지만 다음엔 더 멀리 갈 수 있다"고 느끼게 만드는 것이 재도전 동기의 핵심입니다.

### 차시 설계 점검 질문

- 문제를 푸는 것이 보상으로 이어지는 행동의 도구로 설계되었는가?
- 랜덤 보상이 있고, 기대감과 긴장감이 함께 있는가?
- 최고 보상은 충분히 얻기 어려운가?
- 정답이 유리하되, 정답만이 모든 보상을 보장하지는 않는가?
- 못해도 빈손은 아니고, 운으로 한 단계 더 갈 여지가 있는가?
- 학생이 "다음엔 더 멀리"를 떠올리며 다시 하고 싶어지는가?

## 3차시 매스몬 0 공장 진행 구조 메모

3차시 `매스몬 0 공장`은 `(몇십)×(몇십), (몇십몇)×(몇십)`에서 끝의 0을 처리하는 원리를 연습하는 게임입니다. 학생 화면의 핵심 행동은 항상 `핵심 곱 고르기 -> 붙일 0 개수 고르기` 두 단계입니다.

### 보상/점수 원칙

- 학생에게 실시간 점수 숫자를 크게 보여 주지 않습니다. 문제를 푸는 동안에는 `출하 준비 칸`과 보상 팝업만 보여 주고, 총 생산량은 마지막 결과에서 측정합니다.
- 한 문제의 두 단계를 모두 끝내면 생산 이벤트가 한 번만 일어납니다. 보상은 한 문제당 한 번만 적용됩니다.
- 한 문제 안에서 한 번이라도 틀리면 그 문제는 `불량 처리`로 표시합니다. 이는 일부러 틀려 랜덤 보상을 노리는 흐름을 막기 위한 장치입니다.
- 정답 문제는 랜덤 생산 이벤트를 받습니다. 보통 생산, 대량 생산, 불량 발생, 컨베이어 정지, 즉시 출하, 무지개 부품처럼 기대감과 긴장감이 함께 있어야 합니다.
- 중심 보상은 `생산량/출하 등급` 하나입니다. 별, 코인, 경험치, 도감 점수처럼 별도 보상 체계를 동시에 전면 배치하지 않습니다.
- 최종 결과는 `총 생산량 -> 출하 등급 -> 매스몬 카드` 순서로 보여 줍니다. 매스몬은 결과를 기념하는 카드이며, 문제 풀이 중 보상 체계를 흐리게 만들지 않습니다.

### 권장 진행 흐름

1. 첫 화면: 게임명, 한 줄 목표, 시작 버튼만 우선합니다.
2. 설명 화면: `0을 잠깐 떼기 -> 핵심 곱 계산 -> 0 다시 붙이기`만 짧게 보여 줍니다.
3. 문제 화면: 큰 문제, 컨베이어 보드, 지금 고를 선택지만 남깁니다. 보조 문장이나 장식 이미지는 문제 판단을 흐리면 제거합니다.
4. 1단계: 핵심 곱을 고르면 컨베이어의 첫 칸이 채워집니다.
5. 2단계: 붙일 0의 개수를 고르면 0 스티커와 완성 수가 자동으로 채워집니다.
6. 보상 팝업: 이번 문제의 생산 이벤트를 한 번 보여 주고 출하 준비 칸을 갱신합니다.
7. 결과 화면: 10문제 후 총 생산량을 측정하고 출하 등급과 매스몬 카드를 보여 줍니다.

## Static Game Sharing Via GitHub Pages

학생용 정적 HTML 게임은 이 레포지토리의 GitHub Pages 설정으로 공유한다. 필요한 게임 폴더만 `actions/upload-pages-artifact`와 `actions/deploy-pages`로 배포하고, 공유 전 `curl -I -L <url>`로 실제 URL이 `HTTP 200`인지 확인한다. Pages 실패 시 `gh run view <run-id> --log-failed`로 원인을 먼저 확인한다. checkout 단계에서 submodule 오류가 나면 `.gitmodules`와 `git ls-files -s`로 깨진 `160000` gitlink를 확인하고, 필요 없는 임시 gitlink는 `git rm <path>`로 제거한다.
